FogJack

// 2025 Offensive Tooling Cloud Python 6 min read

Most offensive tooling falls into one of two categories: purpose-built frameworks that are heavily signatured and trivial for defenders to detect, or scattered scripts that require significant glue work before they're useful in an engagement. FogJack started as a solution to that problem, a modular Python based offensive security framework designed to operate on Kali Linux with a focus on flexibility, extensibility, and multi-environment coverage.

This post covers what FogJack is, why I built it, and how it's architected internally. It isn't a full technical disclosure, but enough to explain the design decisions behind it.

Why I Built It

The motivation was straightforward. As companies increasingly migrate their infrastructure to the cloud, the attack surface shifts with them. On-prem perimeter defenses become less relevant when the real assets are sitting in S3 buckets, IAM roles, and managed identities across AWS, Azure, and GCP.

I wanted to understand that shift from an offensive perspective, and the best way to understand something is to build tooling around it. FogJack started as a side project driven by that curiosity, a way to map out what cloud misconfigurations actually look like in practice and how they can be chained together into a full attack path.

It's still a side project. But it's grown into something with real structure, and the cloud exploitation modules in particular reflect genuine research into how attackers move through misconfigured cloud environments.

Architecture

FogJack is built around a modular core. Each capability lives in its own module and exposes a consistent interface to the framework. This means modules can be loaded, swapped, or extended without touching the core. The CLI handles argument parsing and routes execution to the appropriate module.

At a high level, the framework is organized into three layers:

  • Core engine — handles module loading, session state, output formatting, and logging
  • Module layer — individual capability modules that inherit from a base class
  • Interface layer — CLI entry point that routes commands and manages execution flow

The module layer is where the actual offensive capabilities live. Current modules break down as follows:

Network Scanner
Host discovery, port scanning, and service fingerprinting. Wraps raw socket operations and integrates with downstream modules for target selection.
Cloud Enum
Enumerates AWS, Azure, and GCP environments for misconfigurations. Checks IAM role permissions, exposed storage buckets, and overly permissive policies.
Credential Harvesting
Extracts credentials from common storage locations and environment variables across cloud provider CLIs and SDKs.
Post Exploitation
Post-access modules for persistence, lateral movement, and data staging across compromised cloud and on-prem environments.

Multi-Cloud Coverage

One of the more interesting engineering challenges was building consistent cloud enumeration across three providers with fundamentally different APIs and permission models. AWS, Azure, and GCP each have their own SDK, authentication flow, and resource naming conventions.

AWS Attack Vectors

The AWS module targets IAM role abuse as a primary vector. Misconfigured trust policies that allow cross-account assumption, overly permissive inline policies, and roles with wildcard resource access are all common findings. The module also checks for publicly exposed S3 buckets and unencrypted snapshots, which remain surprisingly prevalent. 

# Example: IAM role enumeration flow
fogJack --module cloud-enum --provider aws --check iam-roles
# Enumerates attached policies, inline policies, and trust relationships

Azure Attack Vectors

Azure enumeration focuses on storage account exposure and service principal misconfiguration. Publicly accessible blob containers and storage accounts without access restrictions are common entry points. The module also checks for overprivileged managed identities, which can often be leveraged for lateral movement within a tenant.

GCP Attack Vectors

GCP coverage targets exposed Cloud Storage buckets and service account key abuse. Service accounts with the editor or owner primitive roles at the project level are a reliable escalation path when combined with a foothold inside the environment.

Design Decisions

A few deliberate choices shaped how FogJack is built:

  • Shared session state — modules write to a shared context object so downstream modules can consume upstream findings without re-running enumeration
  • Structured output — all module output is normalized to a consistent format, making it easy to pipe results into reports or other tools
  • Authorization flag — the framework requires an explicit acknowledgment of authorized use before execution, which is a deliberate design choice for responsible tooling
  • Minimal dependencies — core functionality avoids heavy third party libraries where possible to keep the install footprint small on target platforms

Where It's Going

FogJack is still actively in development. Planned additions include deeper Active Directory integration, additional evasion capabilities at the network layer, and expanded post-exploitation modules for cloud-native persistence techniques. The goal is a framework that covers the full attack chain from initial enumeration through post-access operations, all within a single coherent tool.

The source is available on GitHub for those authorized to test in their own environments.
